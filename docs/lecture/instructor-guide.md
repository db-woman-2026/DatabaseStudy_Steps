# 강사용 운영 가이드

`course-plan.md`와 각 `step-N.md`를 운영할 때 쓰는 강사용 체크리스트입니다. 정답을 바로 보여주기보다 먼저 결과를 예상하고 실행 뒤 관찰하게 합니다.

## 1. 수업 전 준비

### 저장소

> Windows 11에서는 [환경 준비](../windows-11.md)를 먼저 확인합니다. `git`, `node`, `npm` 명령은 PowerShell에서도 같습니다. `npm.ps1` 오류가 나면 `npm.cmd`를 사용합니다.

```bash
git status
git branch --show-current
git log --oneline --decorate -3
```

현재 일차와 브랜치가 일치하는지 확인합니다. 학습자에게 뒤 단계 브랜치를 미리 배포하지 않습니다. 누적 브랜치의 목적은 완성 코드를 한꺼번에 보는 것이 아니라, 한 단계에서 바뀐 데이터 모델과 코드만 비교하는 데 있습니다.

### 실행 환경

1~3일차에는 다음을 확인합니다.

```bash
node --version
npm start
```

4~8일차에는 MongoDB 서버 또는 Atlas, `.env`, 안전한 데이터베이스 이름을 추가로 확인합니다.

```bash
cp .env.example .env
npm ci
npm start
```

PowerShell에서는 `.env`를 다음과 같이 준비합니다.

```powershell
Copy-Item .env.example .env
npm.cmd ci
npm.cmd start
```

실습 시작 전에 강사 화면에서 성공 실행 한 번, 의도된 실패 실행 한 번을 준비합니다. 오류 메시지는 수업의 방해물이 아니라 데이터 규칙이 실제로 작동한다는 증거로 다룹니다.

## 2. 60분 블록 운영법

| 구간 | 권장 시간 | 강사의 역할 | 학습자의 역할 |
| --- | ---: | --- | --- |
| 질문과 예측 | 10분 | 실제 상황과 결과 예측 질문 제시 | 개인 예측 작성, 짝과 비교 |
| 개념 설명 | 15분 | 용어를 데이터 예와 연결 | 질문, 반례 제안 |
| 코드 시연 | 10분 | 한 번에 한 변화만 실행 | 실행 전 결과 수/오류 예측 |
| 안내 실습 | 15분 | 막히는 지점 관찰, 힌트 제공 | 직접 입력·실행·기록 |
| 독립 변형 | 5분 | 도메인 또는 조건 변형 제시 | 새 값으로 원리 재현 |
| 확인 | 5분 | 핵심 질문과 다음 블록 연결 | 결과·이유를 한 문장으로 설명 |

학습자 속도가 느리면 독립 변형을 다음 블록 복습으로 이동할 수 있습니다. 개념 설명과 사후 검증은 줄이지 않습니다.

## 3. 설명할 때 지킬 원칙

### 용어와 코드를 항상 연결합니다

“기본키는 고유합니다”에서 끝내지 않습니다. `members.id`의 두 행이 같은 값을 가질 수 없는 이유, 그 ID를 `loans.member_id`가 가리킬 때 얻는 이점을 실제 행으로 확인합니다.

### 행 또는 문서 수를 먼저 묻습니다

쿼리 실행 전에 다음 세 질문을 반복합니다.

1. 입력 데이터는 몇 건인가?
2. 조건을 통과하는 데이터는 몇 건인가?
3. 결과의 한 행 또는 한 문서는 무엇을 나타내는가?

JOIN과 `$unwind`에서 결과 수를 예측하는 습관은 중복과 누락을 찾는 가장 좋은 출발점입니다.

### 실패를 분류합니다

오류가 발생하면 바로 정답을 말하지 않고 다음 범주를 고르게 합니다.

- 터미널/인자 문제
- JavaScript 문법 또는 입력 변환 문제
- 데이터베이스 연결 문제
- 스키마/제약조건 문제
- 조회 조건이 맞지 않는 문제
- 데이터는 정상이나 기대 결과를 잘못 예측한 문제

### 제품 비교를 승패로 만들지 않습니다

RDBMS와 MongoDB 중 무엇이 더 좋다는 결론을 요구하지 않습니다. 같은 요구사항에서도 데이터 일관성, 함께 읽는 범위, 변경 빈도, 배열 성장, 쿼리 패턴에 따라 선택이 달라짐을 반복합니다.

## 4. 질문 은행

### 개념 확인 질문

- 이 값은 한 건을 고유하게 식별하는가, 단지 사람이 읽기 좋은가?
- 이 규칙을 애플리케이션만 검사하면 다른 프로그램이 저장할 때는 어떻게 되는가?
- 이 데이터는 항상 함께 읽히는가? 독립적으로 변경되는가?
- 배열의 원소 수는 최대가 정해져 있는가?
- 이 조건이 빠지면 몇 건이 수정 또는 삭제되는가?
- 처리 건수가 0인 것은 오류인가, 변경할 대상이 없는 정상 결과인가?
- 인덱스가 이 조회에는 도움이 되지만 어떤 비용을 추가하는가?

### 결과 예측 질문

- `LEFT JOIN`의 왼쪽 행은 연결 상대가 없어도 남는가?
- `COUNT(*)`와 `COUNT(books.id)`는 연결된 책이 없는 저자에서 어떻게 다른가?
- `categories: "database"`가 배열 필드와 어떻게 일치하는가?
- `$addToSet`을 같은 값으로 두 번 실행할 때 배열은 어떻게 되며, 같은 update의 `updatedAt` 변경이 `modifiedCount` 해석에 어떤 영향을 주는가?
- 빈 `reviews` 배열에 `$unwind`를 적용하면 그 도서 문서는 결과에 남는가?

### 설계 질문

- 저자 이름이 바뀌었을 때 모든 도서 문서를 갱신해도 되는가?
- 주문 당시 배송 주소는 회원의 현재 주소를 참조해야 하는가, 주문에 복사해야 하는가?
- 리뷰가 백만 개가 될 수 있다면 도서 문서 배열에 모두 넣어도 되는가?
- 대출 생성과 재고 감소 중 하나만 성공하면 어떤 사용자 문제가 생기는가?

## 5. 힌트 제공 단계

학습자가 막혔을 때 완성 코드를 바로 제공하지 않습니다.

1. 오류 메시지에서 핵심 명사와 숫자를 읽게 합니다.
2. 실행한 명령과 현재 브랜치를 확인하게 합니다.
3. 원본 데이터 한 건 또는 테이블 구조를 출력하게 합니다.
4. 복합 조건을 하나씩 제거해 최소 실패 사례를 만듭니다.
5. 비슷한 기준 예제를 문서에서 찾게 합니다.
6. 의사코드 또는 SQL 절반을 제공합니다.
7. 마지막으로 완성 예제와 비교하게 합니다.

힌트를 받은 학습자는 해결 후 “무엇이 원인이었는가?”를 실습 기록지에 남깁니다.

## 6. 짝 활동 운영

짝 활동에서는 한 명이 키보드를 독점하지 않도록 10~15분마다 역할을 바꿉니다.

- Driver: 코드를 입력하고 명령을 실행합니다.
- Navigator: 실행 전 결과를 예측하고 안전 조건을 확인합니다.
- Reviewer: 처리 건수와 사후 조회가 기록되었는지 확인합니다.

2인이라면 Navigator가 Reviewer 역할도 맡습니다. 역할 교대 시 지금까지 바뀐 데이터 상태를 말로 인계합니다.

## 7. 코드 시연 체크리스트

- [ ] 현재 브랜치 이름을 화면에 보였습니다.
- [ ] 실행 전 데이터 상태를 조회했습니다.
- [ ] 바꿀 코드 또는 조건 한 곳만 표시했습니다.
- [ ] 결과 건수를 먼저 예측하게 했습니다.
- [ ] 정상 결과의 반환 객체를 읽었습니다.
- [ ] 실패 결과의 오류 메시지를 숨기지 않았습니다.
- [ ] 실행 후 같은 식별자로 다시 조회했습니다.
- [ ] 왜 그런 결과가 나왔는지 데이터베이스 용어로 설명했습니다.

## 8. 자주 생기는 수업 운영 문제

### 일부 학습자의 데이터 상태가 다릅니다

SQLite 단계는 `npm start`가 현재 단계 전용 파일을 다시 만듭니다. MongoDB CRUD 단계는 `npm start -- seed`로 기준 상태를 복원합니다. 단, seed 전에 개인 실습 데이터가 사라진다는 점을 알립니다.

### 복사한 명령의 따옴표가 다릅니다

문서의 곧은 큰따옴표(`"`)를 사용하게 하고, 제목처럼 공백이 있는 인자를 한 쌍의 따옴표로 감쌉니다. 명령을 통째로 다시 입력하기 전에 `process.argv`에 어떤 배열이 들어올지 칠판에 적습니다.

### MongoDB 연결이 되지 않습니다

애플리케이션 코드를 먼저 고치지 않습니다. 서버 실행 여부, URI, 네트워크 허용, 인증, 데이터베이스 안전 접두사 순으로 분리해 확인합니다.

### 빠른 학습자가 기준 코드를 크게 바꿉니다

기준 실습을 완료하고 결과 기록을 남긴 뒤 별도 커밋 또는 메모에서 확장하게 합니다. 다음 단계 브랜치 전환을 방해할 정도의 큰 변경은 별도 작업 디렉터리에서 진행합니다.

### 느린 학습자가 타이핑에 시간을 모두 씁니다

핵심 코드는 저장소의 기준 코드를 실행하고, 학습자는 조건·값·필드 한두 곳을 직접 바꾸게 합니다. 학습 목표는 긴 코드를 받아쓰는 것이 아니라 결과를 예측하고 데이터 변화를 설명하는 것입니다.

## 9. 평가 피드백 문장 예시

좋은 피드백은 결과뿐 아니라 다음 행동을 알려줍니다.

- “JOIN 결과는 맞습니다. 이제 각 결과 행이 어떤 기준 행을 나타내는지 설명해 보세요.”
- “수정은 성공했습니다. 실행 전 대상 조회와 실행 후 `changes` 확인을 기록에 추가하세요.”
- “포함 모델을 선택한 근거가 조회 편의에만 머물렀습니다. 저자 이름 변경 시 중복 갱신 비용도 비교하세요.”
- “`modifiedCount`가 0인 원인을 오류로 처리했습니다. 이미 같은 값인 경우와 대상을 찾지 못한 경우를 `matchedCount`로 구분하세요.”
- “집계 결과는 맞지만 `$unwind` 뒤 문서 수를 설명하지 못했습니다. 단계별 중간 출력을 만들어 다시 추적하세요.”

## 10. 일차 종료 절차

마지막 15분에는 다음 순서를 지킵니다.

1. 학습자가 오늘의 핵심 개념 세 개를 작성합니다.
2. 가장 유용했던 성공 한 번과 오류 한 번을 기록합니다.
3. 완료 기준을 체크합니다.
4. 3문항 출구 티켓을 제출합니다.
5. 다음 단계에서 달라질 질문을 한 개 남깁니다.
6. `git status`와 데이터 초기화 범위를 확인합니다.

출구 티켓 정답률이 낮은 개념은 다음 날 첫 블록의 변형 예제로 다시 다룹니다.

## 11. 강의 품질 점검

매일 강의 후 강사는 다음을 기록합니다.

| 항목 | 기록할 내용 |
| --- | --- |
| 실제 소요 시간 | 블록별 시작·종료 시각 |
| 막힌 지점 | 명령, 개념, 환경 문제 |
| 생략한 확장 활동 | 다음 날 이동 여부 |
| 오개념 | 반복해서 나온 잘못된 설명 |
| 좋은 학습자 예시 | 다음 기수에서 사용할 설명/반례 |
| 문서 수정 후보 | 모호한 문장, 누락된 예상 결과 |

수업 문서를 개선할 때는 해당 단계 브랜치에서 수정하고 뒤 브랜치로 순차 병합하여 ancestor 구조를 보존합니다.
