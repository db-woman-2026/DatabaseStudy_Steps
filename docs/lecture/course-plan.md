# 데이터베이스 8일 과정

하루 6시간씩 8일 동안 진행합니다. 일차별 설명과 실습, 정답은 해당 브랜치의 `docs/lecture/step-N.md`에서 확인합니다.

## 1. 과정의 성격

이 과정은 데이터베이스 제품의 명령을 암기하는 수업이 아닙니다. 같은 도서 관리 요구사항을 관계형 데이터베이스와 문서 데이터베이스로 각각 표현하면서 다음 질문에 답하는 훈련입니다.

1. 어떤 데이터를 저장해야 하는가?
2. 데이터의 한 건을 어떻게 식별할 것인가?
3. 데이터 사이 관계와 제약을 어디에서 보장할 것인가?
4. 필요한 데이터를 어떤 조건으로 읽을 것인가?
5. 변경이 일부만 반영되지 않도록 어떻게 보호할 것인가?
6. 실행 결과가 의도와 같은지 무엇으로 검증할 것인가?

주 실습 도메인은 도서관의 회원·저자·도서·대출·리뷰입니다. 개념을 특정 도메인에만 가두지 않도록 쇼핑몰, 학교, 예약, 음악 재생 목록, 센서 로그 예제도 함께 비교합니다.

## 2. 대상 학습자와 선수 지식

### 대상

- JavaScript의 변수, 배열, 객체, 함수, 조건문을 한 번 이상 사용한 학습자
- `npm start`와 터미널 명령을 실행할 수 있는 학습자
- 데이터베이스를 처음 배우거나 CRUD만 단편적으로 사용해 본 학습자

### 수업 전에 확인할 선수 지식

- 객체에서 `book.title`, `book.author.name`처럼 속성을 읽을 수 있습니다.
- 배열을 `for ... of`로 순회할 수 있습니다.
- 함수의 인자와 반환값을 구분할 수 있습니다.
- 동기 코드와 `async`/`await` 코드의 모양이 다르다는 것을 알고 있습니다.
- 현재 디렉터리, 파일 경로, 환경 변수의 개념을 대략 이해합니다.

Node.js 자체를 다시 가르치는 과정은 아니지만, 데이터베이스 코드를 읽는 데 필요한 문법은 각 일차 문서에서 짧게 복습합니다.

## 3. 시간 계산 기준

하루는 60분 단위 수업 블록 6개, 총 360분으로 설계합니다. 각 블록은 기본적으로 `도입·설명 15분 + 시연 10분 + 실습 25분 + 확인 10분`으로 운영합니다. 학습 환경에 따라 블록 안의 마지막 10분을 휴식으로 사용할 수 있습니다.

각 일차 문서에는 다음 세 종류의 분량을 모두 제공합니다.

- 필수 경로: 반드시 수행할 설명과 실습
- 확장 경로: 진도가 빠른 학습자를 위한 추가 조건과 변형
- 회복 경로: 오류가 생기거나 진도가 느릴 때 최소 목표를 달성하는 절차

따라서 강사는 학습자 속도에 맞춰 일부 확장 문제를 다음 날 복습으로 이동할 수 있지만, 필수 경로의 개념 확인과 결과 검증은 생략하지 않습니다.

## 6. 단계 사이 학습 연결

과정은 매일 새로운 제품 기능을 나열하지 않고, 이전 날의 질문을 다음 날의 출발점으로 사용합니다.

```text
한 테이블의 중복 문제
  → 여러 테이블과 관계
  → 관계 데이터를 정확히 조회·변경하는 SQL
  → 같은 요구사항을 한 문서에 모으는 MongoDB
  → 문서의 포함/참조와 인덱스 판단
  → 실제 Create/Read 명령
  → Update/Delete와 안전성
  → 배열 문서와 집계, 종합 설계
```

각 일차 종료 시 학습자는 다음 날 질문 하나를 남깁니다. 예를 들어 2일차에는 “JOIN이 많아질수록 읽기 코드는 왜 복잡해지는가?”, 5일차에는 “포함한 배열이 계속 커지면 어떤 문제가 생기는가?”를 기록합니다. 다음 날 첫 블록에서 이 질문을 회수합니다.

## 7. 공통 수업 진행 패턴

모든 실습은 다음 순서를 사용합니다.

1. **예측**: 실행 전에 결과 행/문서 수와 바뀔 값을 적습니다.
2. **실행**: 한 번에 한 조건 또는 한 연산만 변경합니다.
3. **관찰**: 출력, 반환 객체, 오류 메시지를 그대로 기록합니다.
4. **설명**: 결과가 나온 이유를 키·조건·제약·연산자 용어로 설명합니다.
5. **검증**: 같은 대상을 다시 조회해 최종 저장 상태를 확인합니다.
6. **변형**: 값, 조건, 도메인을 바꿔 같은 원리를 재현합니다.

코드가 오류 없이 끝나는 것만으로 완료하지 않습니다. 저장 상태와 처리 건수가 예상과 같은지 확인해야 실습이 끝납니다.

| 트랜잭션 | 재고 감소+대출 생성 | 재고 감소+주문 생성 | 좌석 감소+예약 생성 |
| 중첩 문서 | 도서의 저자 요약 | 주문의 배송지 스냅샷 | 예약의 참석자 요약 |
| 무한 성장 위험 | 리뷰 배열 | 주문 이벤트 배열 | 센서 측정값 배열 |
| 집계 | 카테고리별 재고 | 상태별 매출 | 강좌별 수강 인원 |

변형 예제의 목적은 새 코드를 많이 만드는 것이 아니라, 같은 개념이 데이터 이름이 바뀌어도 적용되는지 확인하는 것입니다.

## 10. 평가 설계

### 매일 형성평가 20점

| 영역 | 점수 | 확인 방법 |
| --- | ---: | --- |
| 용어 설명 | 4 | 핵심 용어 두 개를 예와 함께 설명 |
| 결과 예측 | 4 | SQL/명령 실행 전 행·문서 수 예측 |
| 실습 수행 | 6 | 필수 경로를 오류 처리와 함께 실행 |
| 결과 검증 | 4 | 처리 건수와 사후 조회를 제시 |
| 회고 | 2 | 실수 한 가지와 다음 시도 기록 |

### 3일차 RDBMS 구간평가 30점

- 8점: 요구사항에서 테이블과 키 식별
- 8점: JOIN과 집계 쿼리 작성
- 8점: 안전한 UPDATE/DELETE와 영향 행 수 검증
- 6점: 트랜잭션이 필요한 이유와 실패 시 상태 설명

### 8일차 최종평가 50점

- 10점: 문서 모델과 포함/참조 근거
- 10점: Create/Read 입력·조건·결과 검증
- 10점: Update/Delete 안전 절차와 연산자 선택
- 10점: 집계 파이프라인 단계별 설명
- 10점: 동일 요구사항에서 RDBMS와 MongoDB 선택 근거

정답 코드의 모양이 조금 달라도 데이터 모델, 조건, 결과 검증이 타당하면 인정합니다. 반대로 출력만 우연히 맞고 조건의 의미를 설명하지 못하면 완전한 점수를 주지 않습니다.

## 11. 완료 기준

과정을 수료한 학습자는 도움 없이 다음을 수행할 수 있어야 합니다.

1. 요구사항에서 저장할 데이터, 식별자, 관계, 제약을 찾습니다.
2. 관계형 테이블과 MongoDB 문서 중 하나를 무조건 우월하다고 말하지 않고 접근 패턴에 따라 선택합니다.
3. SQLite에서 테이블을 만들고 prepared statement로 CRUD를 수행합니다.
4. JOIN, 그룹 집계, 트랜잭션의 목적을 실제 데이터 변화로 설명합니다.
5. MongoDB에서 중첩 객체와 배열을 생성·조회·부분 수정·삭제합니다.
6. unique index와 검색 인덱스의 이득과 비용을 설명합니다.
7. 집계 파이프라인에서 각 단계의 입력과 출력을 추적합니다.
8. 변경·삭제 전에 대상을 확인하고, 처리 후 카운트와 재조회로 검증합니다.
9. 정상 입력뿐 아니라 빈 값, 중복 값, 존재하지 않는 ID, 경계값을 시험합니다.
10. 오류 메시지를 지우거나 무시하지 않고 어느 계층에서 실패했는지 분류합니다.

## 12. 환경과 데이터 안전 원칙

- SQLite는 `data/library-step-N.sqlite`처럼 현재 단계 전용 파일만 사용합니다.
- MongoDB는 `MONGODB_DB`가 `database_study_`로 시작하는 강의 전용 데이터베이스일 때만 실행합니다.
- `seed`는 강의용 컬렉션을 초기화하므로 실제 서비스 데이터베이스에서 실행하지 않습니다.
- `.env`는 커밋하지 않고 `.env.example`만 공유합니다.
- `deleteMany({})`의 범위는 현재 단계 전용 컬렉션인지 반드시 확인합니다.
- 수정·삭제는 `사전 조회 → 대상 확인 → 실행 → 처리 건수 → 사후 조회` 순서를 지킵니다.
- 오류 재현을 위해 제약조건을 제거하거나 안전 검사를 우회하지 않습니다.

## 13. 브랜치 학습 규칙

브랜치는 다음 ancestor 순서를 유지합니다.

```text
main -> step-1 -> step-2 -> step-3 -> step-4 -> step-5 -> step-6 -> step-7 -> step-8
```

학습자는 한 번에 현재 일차 브랜치만 사용합니다. `step-3`에서 실습하는 동안 `step-4`의 MongoDB 코드나 문서가 보이지 않아야 합니다. 다음 단계로 이동하기 전에는 수정 중인 파일을 별도로 보관하거나 `git status`를 깨끗하게 정리합니다.

강의 자료 변경도 같은 원칙을 따릅니다.

- 공통 안내는 `main`에서 시작해 뒤 브랜치로 순차 병합합니다.
- `step-N` 전용 자료는 `step-N`에서 시작해 뒤 브랜치로만 순차 병합합니다.
- 앞 단계를 건너뛰어 cherry-pick하거나 같은 변경을 여러 브랜치에 별도 커밋하지 않습니다.
- 병합 후 모든 인접 브랜치에 `git merge-base --is-ancestor` 검사를 수행합니다.
