# 01. 데이터베이스와 RDBMS

1일차 전에 읽을 자료입니다. 단순 파일과 DBMS의 차이, 테이블의 각 요소가 실제 데이터에서 뜻하는 내용을 먼저 정리합니다.

## 1. 데이터, 정보, 데이터베이스

`김민지`, `978-00-0001`, `3`은 맥락 없이 보면 단순한 값입니다. 각각 회원 이름, ISBN, 재고 수량이라는 의미와 연결되면 정보가 됩니다. 데이터베이스는 이 값들을 일정한 구조와 규칙으로 저장하고 다시 사용할 수 있게 합니다.

데이터베이스가 해결해야 하는 일은 단순히 “파일에 기록하기”보다 넓습니다.

- 원하는 조건의 데이터를 빠르게 찾습니다.
- 같은 데이터가 불필요하게 반복되지 않도록 구조화합니다.
- 잘못된 값이나 중복 값을 저장하지 못하게 합니다.
- 여러 변경을 하나의 작업처럼 처리합니다.
- 여러 프로그램이나 사용자가 같은 데이터를 다룰 때 충돌을 줄입니다.
- 장애 뒤에도 저장된 상태를 복구할 수 있게 돕습니다.

## 2. 파일 저장만 사용할 때 생기는 문제

회원 세 명을 JSON 파일에 저장하는 일은 어렵지 않습니다.

```json
[
  { "id": 1, "name": "김민지", "email": "minji@example.com" },
  { "id": 2, "name": "이준호", "email": "junho@example.com" },
  { "id": 3, "name": "박서연", "email": "seoyeon@example.com" }
]
```

하지만 요구사항이 늘어나면 애플리케이션이 직접 해결해야 할 일이 급격히 많아집니다.

### 조건 검색

email이 같은 회원을 찾으려면 파일 전체를 읽고 JSON을 파싱하고 배열을 순회해야 합니다. 데이터가 수백만 건이면 매번 전체 파일을 읽는 방식은 비효율적입니다.

### 중복 방지

두 요청이 거의 동시에 같은 email을 확인하면, 두 요청 모두 “아직 없음”이라고 판단한 뒤 같은 값을 저장할 수 있습니다. 단순한 `if` 검사만으로 동시 요청의 중복을 완전히 막기 어렵습니다.

### 부분 변경

한 회원의 email만 바꾸고 싶어도 파일 전체를 읽고, 객체 하나를 수정하고, 전체 내용을 다시 써야 할 수 있습니다. 쓰는 중 프로그램이 종료되면 파일이 일부만 기록될 위험도 있습니다.

### 관계

회원, 도서, 대출 기록을 각각 파일에 나눴다면 대출의 `memberId`가 실제 회원을 가리키는지 애플리케이션이 항상 확인해야 합니다.

파일 저장이 항상 나쁜 것은 아닙니다. 작은 설정 파일, 한 번 생성하고 거의 바꾸지 않는 결과물, 다른 시스템과 교환할 문서에는 JSON이나 CSV가 적합할 수 있습니다. 핵심은 데이터의 크기, 변경 빈도, 동시 접근, 검색과 제약 요구를 보고 선택하는 것입니다.

## 3. DBMS의 역할

DBMS(Database Management System)는 애플리케이션과 저장 장치 사이에서 데이터 정의, 저장, 조회, 변경, 동시성, 권한, 복구를 담당하는 소프트웨어입니다.

```text
사용자
  ↓
Node.js 애플리케이션
  ↓ SQL 또는 Driver 명령
DBMS
  ↓
메모리·데이터 파일·로그
```

우리가 작성하는 SQL은 디스크의 몇 번째 바이트를 읽으라는 명령이 아닙니다. “재고가 2 이상인 책을 출판 연도 역순으로 달라”처럼 원하는 결과를 선언합니다. DBMS는 인덱스와 실행 계획을 바탕으로 실제 읽기 방법을 선택합니다.

### 데이터베이스와 DBMS를 구분하기

- 데이터베이스: 구조화된 데이터의 모음
- DBMS: 데이터베이스를 관리하는 소프트웨어
- SQLite: 이 과정에서 사용하는 관계형 DBMS이자 라이브러리
- `library-step-1.sqlite`: SQLite가 관리하는 실제 데이터베이스 파일

일상에서는 “SQLite 데이터베이스”처럼 두 용어를 느슨하게 섞어 쓰지만, 설명할 때는 데이터와 관리 소프트웨어를 구분하면 구조를 더 정확히 이해할 수 있습니다.

## 4. 데이터베이스 모델의 종류

데이터를 구조화하는 방식은 하나가 아닙니다.

| 모델 | 대표 구조 | 잘 맞는 질문 예 |
| --- | --- | --- |
| 관계형 | 테이블과 관계 | 주문과 고객을 정확한 키로 연결하고 싶다 |
| 문서형 | 중첩 문서 | 한 상품 상세에 필요한 속성을 한 번에 읽고 싶다 |
| 키-값 | 키와 value | 세션 ID로 세션 값을 매우 빠르게 찾고 싶다 |
| 그래프 | node와 edge | 친구의 친구, 경로, 추천 관계를 탐색하고 싶다 |
| 시계열 | 시간순 측정값 | 센서 값을 시간 구간별로 집계하고 싶다 |

이 과정은 관계형 모델과 문서형 모델을 집중적으로 비교합니다.

## 5. 관계형 데이터베이스의 기본 단위

관계형 데이터베이스는 데이터를 행과 열로 구성된 테이블에 저장합니다.

```text
members
+----+--------+---------------------+---------------------+
| id | name   | email               | joined_at           |
+----+--------+---------------------+---------------------+
|  1 | 김민지 | minji@example.com   | 2026-07-01 09:00:00 |
|  2 | 이준호 | junho@example.com   | 2026-07-02 10:30:00 |
+----+--------+---------------------+---------------------+
```

### 테이블

같은 종류의 데이터를 모은 구조입니다. `members`에는 회원, `books`에는 도서를 저장합니다. 서로 의미가 다른 회원과 도서를 한 테이블에 섞지 않습니다.

### 행

한 개의 데이터 또는 한 사실을 나타냅니다. 위 표의 첫 번째 행은 회원 김민지 한 명입니다. record 또는 tuple이라는 용어도 사용합니다.

### 열

모든 행이 공유하는 속성입니다. 열에는 이름, 데이터 타입, 기본값, 필수 여부 같은 규칙이 있습니다.

### cell/value

특정 행과 열이 만나는 값입니다. `id=1` 행의 `email` 값은 `minji@example.com`입니다.

### 스키마와 인스턴스

- 스키마: 테이블·열·타입·제약조건처럼 데이터 구조를 정의한 설계
- 인스턴스/state: 특정 시점에 실제로 저장된 행들의 상태

`CREATE TABLE`을 바꾸는 것은 스키마를 바꾸는 일이고, `INSERT`로 회원을 추가하는 것은 현재 인스턴스를 바꾸는 일입니다.

## 6. JavaScript 객체와 행 비교

```js
const member = {
  id: 1,
  name: "김민지",
  email: "minji@example.com",
}
```

객체 하나와 행 하나는 “한 대상의 여러 속성을 이름과 값으로 묶는다”는 점에서 비슷합니다. 그러나 차이도 있습니다.

| JavaScript 객체 | 관계형 테이블의 행 |
| --- | --- |
| 객체마다 속성이 달라도 실행 가능 | 테이블 스키마에 정의된 열을 따름 |
| 메모리에 있다가 프로세스 종료 시 사라질 수 있음 | DB 파일에 지속적으로 저장 가능 |
| 값 검증을 코드로 구현 | DB 제약조건으로도 검증 가능 |
| 다른 객체와 참조가 런타임 메모리 주소일 수 있음 | 키 값으로 다른 테이블 행을 참조 |

객체와 행을 완전히 같은 것으로 생각하면 안 되지만, 객체를 SQL 인자에 전달하고 조회 행을 객체로 받는 흐름을 이해하는 데 이 비교가 유용합니다.

## 7. 키의 역할

이름은 사람을 식별하기에 충분하지 않습니다. 동명이인이 있을 수 있고 이름이 바뀔 수도 있습니다. 데이터베이스는 각 행을 안정적으로 구분할 키가 필요합니다.

### 기본 키

테이블의 각 행을 고유하게 구분하는 대표 키입니다.

```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
```

`id`는 중복되지 않으며, 다른 테이블이 이 회원을 가리킬 때 사용할 수 있습니다.

### 자연키와 대체키

- 자연키: 업무에서 이미 의미가 있는 값. 예: ISBN, 주민등록번호, 상품 SKU
- 대체키: 데이터베이스에서 식별만을 위해 만든 값. 예: 자동 증가 `id`

ISBN도 도서를 식별할 수 있지만 입력 형식 변경이나 정책 변화에 영향을 받을 수 있습니다. 실무에서는 내부 관계용 대체키와 업무상 고유한 값의 `UNIQUE` 제약을 함께 사용하는 경우가 많습니다.

### 좋은 키를 고르는 질문

1. 정말 모든 행에서 값이 존재하는가?
2. 중복될 가능성이 없는가?
3. 시간이 지나도 값이 바뀌지 않는가?
4. 너무 길거나 민감한 정보는 아닌가?

## 8. 데이터 타입과 NULL

SQLite의 타입 체계는 다른 RDBMS보다 유연하지만, 열의 의도를 타입으로 표현하는 습관은 중요합니다.

| 선언 | 주 용도 | 예 |
| --- | --- | --- |
| `INTEGER` | 정수, ID, 개수 | `3`, `2026` |
| `REAL` | 실수 | `4.5` |
| `TEXT` | 문자열, ISO 날짜 문자열 | `"김민지"` |
| `BLOB` | 바이너리 데이터 | 이미지 바이트 |

`NULL`은 빈 문자열이나 숫자 0이 아닙니다. “값이 없거나 알려지지 않음”을 나타냅니다.

- `phone = ''`: 전화번호 필드는 있지만 빈 문자열을 저장함
- `phone IS NULL`: 전화번호 값이 없음
- `stock = 0`: 재고 수량을 알고 있으며 0권임
- `stock IS NULL`: 재고 수량을 알 수 없음

SQL에서는 `column = NULL`이 아니라 `column IS NULL`을 사용합니다.

## 9. 제약조건은 데이터 규칙이다

```sql
CREATE TABLE members (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE,
  joined_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
)
```

### `NOT NULL`

필수 값이 빠진 행을 막습니다. 회원 이름이나 email이 없는 상태가 업무적으로 허용되지 않는다는 규칙입니다.

### `UNIQUE`

두 행이 같은 email을 갖지 못하게 합니다. 애플리케이션의 사전 중복 확인과 별개로 데이터베이스가 최종 경계에서 보장합니다.

### `DEFAULT`

값을 생략했을 때 사용할 기본값입니다. 기본값은 “아무 값이나 넣어도 됨”이 아니라, 생략할 때 합리적인 업무 규칙이 있어야 합니다.

### `CHECK`

```sql
stock INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0)
```

값의 범위를 제한합니다. 재고가 음수가 되는 잘못된 상태를 데이터베이스에서 막을 수 있습니다.

제약조건 오류는 프로그램이 망가졌다는 뜻만은 아닙니다. 저장하려던 데이터가 정의된 규칙을 위반했다는 중요한 피드백입니다.

## 10. SQLite의 특징

SQLite는 별도 서버 프로세스 없이 애플리케이션이 라이브러리로 불러와 데이터베이스 파일을 직접 다룹니다.

### 학습에 적합한 이유

- 서버 설치와 계정 설정 없이 SQL에 집중할 수 있습니다.
- 데이터베이스 하나가 파일 하나여서 실습 상태를 초기화하기 쉽습니다.
- 표준 SQL의 핵심인 DDL, CRUD, JOIN, 집계, 트랜잭션을 연습할 수 있습니다.
- Node.js 22.13 이상의 `node:sqlite` 모듈로 별도 패키지 없이 실행할 수 있습니다.

### 서버형 RDBMS와 다른 점

- 많은 원격 클라이언트가 동시에 접근하는 서버 구조가 아닙니다.
- 타입 처리와 일부 SQL 문법이 PostgreSQL, MySQL 등과 다를 수 있습니다.
- 운영 규모, 동시 쓰기, 권한 관리 요구에 따라 서버형 DBMS가 더 적합할 수 있습니다.

SQLite로 배운 테이블, 키, 제약, SQL, 트랜잭션 개념은 다른 RDBMS로 옮겨갈 수 있지만 제품별 문법과 운영 특성은 별도로 확인해야 합니다.

## 11. 이 저장소에서의 연결 흐름

```js
const { DatabaseSync } = require("node:sqlite")
const database = new DatabaseSync(databasePath)

database.exec("PRAGMA foreign_keys = ON")
// 테이블 생성, 행 생성, 조회
database.close()
```

### 연결

`DatabaseSync` 객체는 SQL을 실행할 데이터베이스 연결입니다. 경로에 파일이 없으면 SQLite가 새 파일을 만듭니다.

### `exec()`

여러 SQL 문장 또는 반환 행을 직접 사용하지 않는 DDL을 실행할 때 편리합니다.

### `prepare()`

값이 달라지는 SQL을 미리 준비합니다. `?` 자리에 값을 별도로 전달하면 SQL 문자열과 입력값을 분리할 수 있습니다.

### `close()`

작업이 끝났음을 연결에 알리고 자원을 정리합니다. 오류가 있더라도 닫을 수 있도록 실제 서비스에서는 정리 경로를 신중히 설계합니다.

## 12. 도메인 변형 예제

### 쇼핑몰 상품

```text
products(id, sku, name, price, stock)
```

- `id`: 내부 기본 키
- `sku`: 업무상 고유 상품 코드, UNIQUE 후보
- `price`: 음수가 아니어야 함
- `stock`: 0 이상의 정수

### 학교 학생

```text
students(id, student_number, name, admitted_year)
```

학생 이름은 중복될 수 있지만 학번은 중복되면 안 됩니다. 이름을 기본 키로 선택하면 안 되는 이유를 설명할 수 있어야 합니다.

### 공연 좌석

```text
seats(id, performance_id, seat_code, status)
```

같은 공연 안에서 좌석 코드는 고유해야 하지만 다른 공연에서는 `A-01`이 다시 등장할 수 있습니다. 나중에 복합 UNIQUE 제약을 배울 수 있는 예입니다.

5. 애플리케이션이 email 중복을 검사해도 `UNIQUE`가 필요한 이유는 무엇인가요?
6. SQLite가 학습에 편리한 이유와 운영 환경에서 검토할 한계를 하나씩 적으세요.
7. 회원, 도서, 대출을 한 테이블에 모두 저장하면 어떤 값이 반복될지 예상하세요.

<details>
<summary>개념 확인 해설</summary>

1. 파일도 넓은 의미의 데이터 모음일 수 있지만, DBMS는 구조 정의, 검색, 제약, 동시성, 복구 등을 관리하는 소프트웨어입니다.
2. 스키마는 `id INTEGER`, `email TEXT UNIQUE` 같은 구조이고 인스턴스는 현재 저장된 김민지·이준호 행입니다.
3. 두 사람을 구분할 수 없고 이름 변경 시 참조 값도 바뀌며, 대출 기록이 어느 사람의 것인지 모호해집니다.
4. 0은 수량을 알고 있으며 없다는 값이고, NULL은 수량이 없거나 알려지지 않았다는 상태입니다.
5. 동시 요청이나 다른 프로그램의 저장을 애플리케이션 사전 검사만으로 완전히 막지 못할 수 있기 때문입니다.
6. 서버 설치 없이 파일 하나로 SQL을 연습할 수 있습니다. 반면 대규모 동시 쓰기나 서버 권한 관리 요구에는 다른 선택이 필요할 수 있습니다.
7. 한 회원이 여러 권을 빌릴 때 회원 이름/email이 반복되고, 같은 책이 여러 번 대출되면 제목·저자도 반복됩니다.

</details>
